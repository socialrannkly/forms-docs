---
title: "Generate Private Link"
api: "POST /external-api/v1/forms/{formId}/links"
description: "Generate a private form link with optional customer data, prefill values, and access controls."
---

This is the primary endpoint of the Rannkly Forms API. It creates a unique, tokenized link to a **private form** that can be shared with a specific customer.

## Request

### Headers

| Header | Value | Required |
|---|---|---|
| `X-API-Key` | Your API key (`rfk_...`) | Yes |
| `X-Team-Id` | Numeric team ID (required for global API keys) | Conditional |
| `Content-Type` | `application/json` | Yes |

### Path Parameters

<ParamField path="formId" type="string" required>
  The MongoDB ObjectId of the form. Must be a 24-character hex string.

  You can find this in the form URL in your dashboard: `https://forms.rannkly.com/forms/{formId}/edit`
</ParamField>

### Body Parameters

<ParamField body="customer" type="object">
  Customer information to attach to this link. If an email is provided, an existing contact with the same email will be updated; otherwise, a new contact is created.

  <Expandable title="Customer Properties">
    <ParamField body="email" type="string">
      Customer email address. Used as the unique identifier to find/update existing contacts.
    </ParamField>
    <ParamField body="name" type="string">
      Customer full name. Max 255 characters.
    </ParamField>
    <ParamField body="phone" type="string">
      Customer phone number. Max 50 characters.
    </ParamField>
    <ParamField body="companyName" type="string">
      Customer company name. Max 255 characters.
    </ParamField>
    <ParamField body="properties" type="array">
      Custom key-value properties to store on the contact.

      ```json
      [
        { "key": "department", "value": "Engineering" },
        { "key": "plan", "value": "Enterprise" }
      ]
      ```

      - `key`: 1–100 characters
      - `value`: up to 1,000 characters
    </ParamField>
  </Expandable>
</ParamField>

<ParamField body="prefillData" type="object">
  Key-value pairs to prefill form fields when the link is opened. Keys should match your form's question IDs (e.g., `q-1770634598292`).

  You can find question IDs using the [Get Form by Slug](/api-reference/forms/get-by-slug) endpoint.

  For simple questions, the value is a string. For composite questions like `contact_info`, the value is an object with sub-field keys:

  ```json
  {
    "q-1770634598292": "12345",
    "q-1770634652323": "John",
    "q-1770634793939": "john@example.com",
    "q-1770635000000": {
      "firstName": "John",
      "lastName": "Doe",
      "email": "john@example.com",
      "phoneNumber": "555-0123"
    }
  }
  ```
</ParamField>

<ParamField body="settings" type="object">
  Per-link overrides for form behavior. These settings are stored on the link token and also applied globally to the form.

  <Expandable title="Settings Properties">
    <ParamField body="prefillEditable" type="boolean" default="true">
      Global default that controls whether **all** prefilled fields can be edited by the respondent.

      - `true` (default) — respondent can modify prefilled values
      - `false` — all prefilled fields are locked as read-only

      Use this when you want a single rule for every prefilled field. To control editability on a per-question basis, use `prefillEditableOverrides` instead.
    </ParamField>
    <ParamField body="prefillEditableOverrides" type="object">
      Per-question editable overrides. Keys are question IDs (same IDs used in `prefillData`). Each override takes priority over the global `prefillEditable` setting for that specific question.

      **Value types:**
      - **`boolean`** — controls the entire question (e.g., `"q-123": false` makes the question read-only)
      - **`object`** — controls individual sub-fields of composite questions like `contact_info` (e.g., `"q-456": { "email": false, "firstName": true }`)

      ```json
      {
        "q-1770634598292": false,
        "q-1770634652323": true,
        "q-1770634793939": {
          "email": false,
          "firstName": true,
          "phoneNumber": false
        }
      }
      ```

      <Info>
        **Priority chain (highest to lowest):**
        1. Per-field override (e.g., `"email": false` inside a composite question)
        2. Per-question override (e.g., `"q-123": false`)
        3. Global `prefillEditable` setting

        Fields not listed in `prefillEditableOverrides` inherit the global `prefillEditable` value.
      </Info>
    </ParamField>
    <ParamField body="cooldown" type="object">
      Cooldown configuration that controls what happens when a respondent tries to access the form again after submitting.

      The cooldown settings are applied **both** globally to the form (affecting all accesses) **and** stored on the link token (for per-link override awareness).

      <Expandable title="Cooldown Properties">
        <ParamField body="enabled" type="boolean" required>
          Enable or disable cooldown for the form.
        </ParamField>
        <ParamField body="duration" type="integer">
          Cooldown duration value. Must be a positive integer. Defaults to `24` if not specified.
        </ParamField>
        <ParamField body="durationUnit" type="string">
          Unit of the cooldown duration. One of: `minutes`, `hours`, `days`. Defaults to `hours`.
        </ParamField>
        <ParamField body="action" type="string">
          What happens when a user tries to access the form during cooldown. One of:

          - `block` — completely block the submission (default)
          - `edit_only` — allow editing the previous submission; the form is prefilled with the respondent's previous answers
          - `view_only` — show the form as read-only with previous answers
        </ParamField>
      </Expandable>
    </ParamField>
  </Expandable>
</ParamField>

<ParamField body="expiresAt" type="string">
  ISO 8601 datetime string with timezone offset. The link will stop working after this time.

  **Example:** `"2026-03-01T00:00:00Z"` or `"2026-03-01T05:30:00+05:30"`

  <Warning>
    The date must include a timezone offset (e.g., `Z` or `+05:30`). Plain dates like `"2026-03-01"` will fail validation.
  </Warning>
</ParamField>

<ParamField body="maxUsages" type="integer">
  Maximum number of times the link can be used to **submit** the form. Must be a positive integer.

  Once this limit is reached, the link becomes inactive and any attempt to open it will return an "Invalid or expired access token" error.

  <Note>
    `maxUsages` counts form **submissions**, not page views. The respondent can open and view the form multiple times without consuming a usage. A usage is only counted when the form is successfully submitted.
  </Note>
</ParamField>

<ParamField body="label" type="string">
  An internal label for this link (e.g., "Feedback link for John Doe"). Max 255 characters. Not visible to the form respondent.
</ParamField>

## How It Works

<Steps>
  <Step title="Validates form ownership">
    The API verifies that the form exists, belongs to your team, and hasn't been deleted.
  </Step>
  <Step title="Applies form settings (if provided)">
    If `cooldown` or `prefillEditable` are provided inside `settings`, the form's global settings are updated **and** the overrides are stored on the link token for per-link awareness. Per-question overrides (`prefillEditableOverrides`) are stored exclusively on the link token and do not modify the global form settings.
  </Step>
  <Step title="Creates or updates the contact">
    If `customer` data is provided, the API looks up an existing contact by email (within your team). If found, the contact is updated; otherwise, a new contact is created with `source: "api"`.
  </Step>
  <Step title="Generates a secure token">
    A cryptographically random token (`tkn_` + 64 hex characters) is generated and stored in the database along with the prefill data, editable overrides, cooldown settings, and form configuration.
  </Step>
  <Step title="Returns the private URL">
    The form URL is constructed as `{baseUrl}/f/{slug}?token={token}` and returned to you.
  </Step>
</Steps>

## Prefill Editable Behavior

When `prefillData` is provided, the editable state of each prefilled field is determined by the following priority chain:

| Priority | Source | Scope | Example |
|---|---|---|---|
| 1 (highest) | `prefillEditableOverrides` with field-level object | Individual sub-field of a composite question | `{ "q-123": { "email": false } }` |
| 2 | `prefillEditableOverrides` with boolean | Entire question | `{ "q-123": false }` |
| 3 (lowest) | `prefillEditable` | All prefilled fields | `true` or `false` |

<AccordionGroup>
  <Accordion title="All prefilled fields editable (default)">
    Don't pass `settings` at all, or set `prefillEditable` to `true`. The respondent can modify every prefilled value.

    ```json
    {
      "prefillData": { "q-name": "John", "q-email": "john@example.com" },
      "settings": { "prefillEditable": true }
    }
    ```
  </Accordion>
  <Accordion title="All prefilled fields read-only">
    Set `prefillEditable` to `false`. Every prefilled field is locked.

    ```json
    {
      "prefillData": { "q-name": "John", "q-email": "john@example.com" },
      "settings": { "prefillEditable": false }
    }
    ```
  </Accordion>
  <Accordion title="Mixed: some read-only, some editable">
    Set `prefillEditable` to the default you want for most fields, then override specific questions via `prefillEditableOverrides`.

    ```json
    {
      "prefillData": {
        "q-order-id": "ORD-98765",
        "q-name": "John",
        "q-email": "john@example.com"
      },
      "settings": {
        "prefillEditable": true,
        "prefillEditableOverrides": {
          "q-order-id": false,
          "q-email": false
        }
      }
    }
    ```

    Result: order ID and email are read-only; name is editable.
  </Accordion>
  <Accordion title="Composite question: per-sub-field control">
    For `contact_info` questions, pass an object to control each sub-field individually.

    ```json
    {
      "prefillData": {
        "q-contact": {
          "firstName": "Jane",
          "email": "jane@example.com",
          "phoneNumber": "555-0123"
        }
      },
      "settings": {
        "prefillEditable": true,
        "prefillEditableOverrides": {
          "q-contact": {
            "email": false,
            "firstName": true,
            "phoneNumber": true
          }
        }
      }
    }
    ```

    Result: email is read-only; first name and phone are editable.
  </Accordion>
</AccordionGroup>

## Cooldown Behavior

When cooldown is enabled, here's what happens during the respondent's interaction:

<Steps>
  <Step title="First visit">
    The respondent opens the link, sees the form (with prefilled data if provided), and submits their response. This counts as one usage toward `maxUsages`.
  </Step>
  <Step title="Re-opens the link during cooldown">
    Depending on the `action` setting:

    - **`block`** — The form submission is rejected with a cooldown message showing remaining time.
    - **`edit_only`** — The form loads in **edit mode** with an "Editing your previous response" banner. The respondent's previous answers are prefilled and they can modify and resubmit. Each resubmission counts toward `maxUsages`.
    - **`view_only`** — The form is shown as read-only with the respondent's previous answers. No resubmission is allowed.
  </Step>
  <Step title="After cooldown expires">
    The respondent can submit a fresh response (if `maxUsages` and `expiresAt` allow).
  </Step>
</Steps>

<Info>
  The cooldown identity method defaults to `email` — the system matches the respondent by the email address in their submission. The form must include an email-type question for this to work reliably.
</Info>

## Access Controls

The link respects three access control mechanisms that work together:

| Control | Description | Checked on |
|---|---|---|
| **`expiresAt`** | Link stops working after this datetime | Every page load |
| **`maxUsages`** | Link stops working after N successful submissions | Every page load |
| **Cooldown** | Controls resubmission behavior during the cooldown window | On form submission and form load |

<Note>
  Opening the form link (page view) does **not** count toward `maxUsages`. Only successful form submissions increment the usage counter.
</Note>
